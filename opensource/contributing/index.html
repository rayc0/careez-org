Contributing

How to contribute
to CareEZ

Bug reports, translations, clinical accuracy review, and docs all welcome. This guide covers the PR process, commit format, and test requirements.

Getting Started

PR process — step by step

  • 1
    Fork and clone Fork the relevant repo on GitHub, then clone your fork. Keep main in sync with upstream — git fetch upstream && git rebase upstream/main before starting any branch.
  • 2
    Create a feature branch Branch from main. Name it type/short-description — e.g. fix/tagalog-level-3-label or feat/vietnamese-delivery.
  • 3
    Make changes with passing tests All tests must pass before opening a PR. For clinical content changes, include a note on which IDDSI levels or clinical rules are affected and link the relevant IDDSI 2019 framework reference.
  • 4
    Open a PR against main Fill in the PR template. One reviewer required for non-clinical changes; two reviewers (one must be a clinical contributor) for any change touching triage logic, IDDSI levels, or escalation rules.
  • 5
    Address review feedback, then squash-merge PRs are merged via squash to keep the commit history clean. The squash commit title follows the commit format below.

Clinical changes require extra care. Any PR that modifies triage logic, IDDSI level classification, aspiration risk scoring, or escalation thresholds must include a Clinical Impact section in the PR description. Tag @careez-clinical for mandatory review.

Commit Format

Conventional Commits — required

All commits must follow Conventional Commits 1.0. The format is <type>(<scope>): <description>. 

# Types
feat      # new feature or capability
fix       # bug fix
docs      # documentation only
i18n      # translation or locale change
clinical  # clinical content, triage logic, IDDSI rules
refactor  # code change — no behaviour change
test      # adding or fixing tests
chore     # build, deps, config — no production code

# Scope (optional but encouraged)
api       # Cloudflare Worker / API layer
assessor  # care-food-assessor package
screener  # dysphagia-screener package
recipe    # recipe-modifier package
cantonese | tagalog | bahasa | vietnamese | thai

# Examples
feat(tagalog): add IDDSI level 4 descriptor translations
fix(assessor): correct minced-moist boundary check for level 5
clinical(screener): update aspiration risk threshold per FEES 2024 study
i18n(vietnamese): initial locale scaffolding
docs: add IDDSI framework reference links to README

Breaking changes: append ! after the type and add a BREAKING CHANGE: footer in the commit body.

Test Requirements

What must pass before merging

General (all PRs)

  • CI passes — npm test (or equivalent) exits 0
  • No new TypeScript errors — tsc --noEmit clean
  • Lint passes — eslint with project config
  • No regression in existing test suite (snapshot or unit)

Clinical content changes

  • IDDSI level classification tests updated to cover the changed logic
  • All 7 IDDSI levels tested for the affected food/liquid path
  • Escalation rule tests pass with the new threshold
  • PR description includes a Clinical Impact section citing the source (IDDSI 2019, FEES study, HKCSS 護食標準, etc.)
  • Tagged @careez-clinical — two-reviewer approval required

Translation / i18n PRs

  • All locale keys present — no missing keys vs. the en base
  • Clinical terminology reviewed by a native speaker with care-sector background where possible — note in PR if reviewed
  • Rendered output spot-checked for character encoding (Cantonese, Tagalog, Bahasa diacritics)

API / Worker changes

  • Wrangler local dev passes — wrangler dev starts without errors
  • Affected API endpoints have integration tests
  • No secrets or credentials committed — CI secret-scan must pass
Code of Conduct

Contributor Covenant 2.1

CareEZ repositories are governed by the Contributor Covenant 2.1. All contributors, maintainers, and community members are expected to uphold it.

Instances of unacceptable behaviour may be reported to hello@careez.org. All reports are reviewed confidentially. Maintainers are obligated to respect the privacy and security of reporters.
